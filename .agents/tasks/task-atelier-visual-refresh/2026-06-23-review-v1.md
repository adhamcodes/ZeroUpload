# ATELIER visual refresh: oklch tokens, dark mode, component upgrades

This change replaces the "Quiet Luxury" hex-based design system with "ATELIER", built on oklch color tokens organized into Porcelain (light) and Vault (dark) palettes. It swaps Inter for Geist/Geist Mono, introduces a brass accent color, adds a dark mode toggle with localStorage persistence, builds out FormatPicker and ToolGrid as new components, and wires quick-chips and tool icons across all hub pages. The conversion engines and scripts are untouched.

**Watch for:** Dark mode token overrides use bare CSS custom properties outside `@theme`, which in Tailwind v4 means utilities like `bg-canvas` resolve to the `@theme`-registered value and may not pick up the `[data-theme="dark"]` override (likely). The `prefersReducedMotion` check in Converter.tsx is captured once at component mount and never updates (confirmed). The `hover-lift` utility class is defined in global.css but never referenced in the actual component markup, which uses inline Tailwind hover utilities instead — so the reduced-motion rule doesn't cover card animations (confirmed).

---

## High-level view

Tailwind v4's `@theme` block compiles token values into the generated CSS. A `[data-theme="dark"]` selector that re-declares the same custom properties at runtime should cascade correctly (Tailwind v4 does emit `var()` references for `@theme` tokens), but this needs build-output verification. If utilities inline the resolved color rather than referencing the variable, dark mode will have no effect on any utility-class-driven color. The `dark:` variant is also not wired — future additions of `dark:bg-surface` would fail silently.

The Header's dark mode script runs after first paint. Users with a stored dark preference or `prefers-color-scheme: dark` will see a flash of the Porcelain palette before Vault applies. A blocking `<script>` in `<head>` would eliminate this.

ToolGrid and ConversionGrid implement hover-lift with inline Tailwind classes (`hover:-translate-y-0.5`) rather than the `.hover-lift` utility class defined in global.css. The `prefers-reduced-motion` rule in global.css only disables `.hover-lift` transitions, so reduced-motion users still see card animations. The unused `.hover-lift` class is dead code.

Geist and Geist Mono are loaded via a Google Fonts URL, but Geist is not officially on Google Fonts — it's distributed via Vercel's CDN. The URL may 404, falling back to system fonts silently.

The Footer uses `hover:text-brass` on the brand name. Brass on the light canvas (~3.2:1 contrast) fails WCAG AA for normal-sized text.

## Dark mode token strategy and Tailwind v4 compatibility

Tailwind v4's `@theme` block defines design tokens that get compiled into utility classes. The critical question is whether the generated `.bg-canvas` class contains `background-color: oklch(96% 0.01 80)` (a static value) or `background-color: var(--color-canvas)` (a variable reference). If the former, then the `[data-theme="dark"]` selector that overrides `--color-canvas` at runtime will have no visible effect — every Tailwind utility would remain frozen at its light-mode value.

Tailwind v4 does generate `var()` references for `@theme` tokens in practice, so this likely works. But the approach has a second subtlety: the `[data-theme="dark"]` block lives outside `@theme`, so Tailwind doesn't know about the dark palette. This means Tailwind's built-in `dark:` variant won't work if someone adds `dark:bg-surface` — the token isn't registered in Tailwind's dark context. For now the codebase doesn't use the `dark:` variant anywhere, so this is a future footgun rather than a current bug.

## Flash of wrong theme (FOWT)

The theme initialization script lives inside `Header.astro`, which means it executes after the full `<body>` has rendered. A user who has `theme: "dark"` in localStorage will see the Porcelain palette for a frame before the Vault palette applies. The fix is a small blocking script in `<head>` (inside Layout.astro) that reads localStorage and sets `data-theme` before any paint occurs:

```html
<script>
  (function() {
    try {
      var t = localStorage.getItem('theme');
      if (t) document.documentElement.setAttribute('data-theme', t);
      else if (matchMedia('(prefers-color-scheme:dark)').matches)
        document.documentElement.setAttribute('data-theme', 'dark');
    } catch(e) {}
  })();
</script>
```

## Reduced motion gaps in ToolGrid and ConversionGrid

Both components implement hover-lift with inline Tailwind utilities:

```html
class="... hover:-translate-y-0.5 hover:shadow-[var(--shadow-lifted)]"
```

The global.css `@media (prefers-reduced-motion: reduce)` rule targets only `.hover-lift` and `.hover-lift:hover`. Since neither component uses that class, users who request reduced motion will still see the card translation and shadow change on hover. The fix is either switching to the `.hover-lift` class (which would also remove the dead code), or prefixing the inline utilities with `motion-safe:` (e.g., `motion-safe:hover:-translate-y-0.5`).

The Converter.tsx `prefersReducedMotion` approach is more targeted — it conditionally renders the glow/shimmer elements. But `getReducedMotion()` is called in the component body outside any effect or state. Since there's no state trigger when the media query changes, the component never re-renders to pick up the new value. A `useState` + `useEffect` with a `matchMedia('(prefers-reduced-motion: reduce)')` listener would make it reactive. Note that the CSS-level `[style*="animation"]` selector in the reduced-motion media query acts as a safety net for the animations even if the React value is stale.

## Font loading: Geist availability on Google Fonts

The Layout.astro loads Geist via `https://fonts.googleapis.com/css2?family=Geist:wght@400;500;600&family=Geist+Mono:wght@400;500&display=swap`. As of late 2024, Geist is not part of the official Google Fonts catalog — it's a Vercel-published typeface distributed at `https://cdn.vercel.com/geist/font/`. If the Google Fonts URL fails, the font-stack falls back gracefully to `ui-sans-serif, system-ui`, but the specific Geist look won't render. Options: verify the URL works in a browser, self-host the woff2 files in `public/fonts/`, or use Vercel's CDN directly.

## Mobile menu accessibility gap

The hamburger button in Header.astro toggles the mobile menu's visibility via `classList.toggle('hidden')`, but does not:
- Set `aria-expanded="true"/"false"` on the button
- Set `aria-hidden` on the menu panel
- Trap focus inside the open menu
- Close on Escape key

Screen readers will not announce whether the menu is open or closed.
