# Category UX hub pages with FormatPicker and ToolGrid

Four hub pages (/pdf, /image-converter, /audio-converter, /text) now funnel visitors through a "From -> To" format picker into existing per-pair SEO pages. A new `FormatPicker.tsx` React island handles selection and navigation; a new `ToolGrid.astro` advertises coming-soon tools alongside the handful of live conversion links. Navigation (Header, Footer) is updated, and `/document-converter` becomes a meta-refresh redirect to `/pdf`.

The implementation is consistent with the Quiet Luxury brand system: Fraunces headings, Inter body, green accent, `--radius-xl` cards, `--shadow-luxe` hover states, privacy messaging on every page, and JSON-LD CollectionPage structured data. Watch for: the FormatPicker on /text navigates to non-existent pages since no `*-to-txt` routes are built (confirmed), the `<select>` labels lack `htmlFor`/`id` pairings degrading accessibility (confirmed), and the Privacy nav link was removed from the header without a replacement (confirmed).

## High-level view

The FormatPicker is a self-contained React island that uses `client:load` on every hub page. Its core UX flaw is that it doesn't validate whether a chosen from-to pair actually exists as a built page. The image, audio, and PDF hubs happen to be safe because `conversions.ts` generates all the pairs those dropdowns can produce. The /text hub is not safe: its sole target is "txt" and no `*-to-txt` pages exist, so every Convert click is a guaranteed 404.

The ToolGrid component mirrors ConversionGrid's card style and adds a coming-soon badge pattern with reduced opacity and no link, but the cards use `<p>` instead of `<h3>` for tool names, degrading screen-reader heading navigation.

Each hub page follows an identical structure (hero with privacy badge, format picker, tool grid, conversion grid) and carries proper meta/SEO (title, description, canonical via Layout, JSON-LD). The old `/document-converter` is a clean redirect with `noindex` and canonical pointing to `/pdf`.

The Header drops the `/#privacy` link entirely to make room for "Text", removing the only top-level path to the homepage privacy section.

## FormatPicker navigation to dead routes

The picker's `handleConvert` constructs a URL from the raw `from` and `to` IDs: `/${from}-to-${to}`. It guards against `from === to` but nothing else.

The **PDF hub** sources include jpg, jpeg, png, webp and the targets include pdf, jpg, png. Every combination is generated by `conversions.ts` (`img2pdf` for images-to-pdf, `pdf2img` for pdf-to-jpg/png). Safe.

The **image hub** sources include heic, svg, gif, bmp — all have `canDecode: true` in `conversions.ts` and all four target formats have `canEncode: true`. Every pair is generated. Safe.

The **audio hub** uses a single `formats` array for both dropdowns and `conversions.ts` generates every audio-to-audio pair. Safe.

The **/text hub**: sources are jpg, png, pdf; the sole target is `txt`. No `*-to-txt` conversion exists in `conversions.ts`, no `[from]-to-[to].astro` page will be generated for those slugs. The button is never disabled (from !== to is always true when target is "txt" and source is anything else), so the user hits a broken route every time.

```tsx
// FormatPicker.tsx, line 27
function handleConvert() {
  if (from && to && from !== to) {
    window.location.href = `/${from}-to-${to}`;
  }
}
```

Mitigation options: disable the picker entirely on /text (add a `disabled` prop), or have the picker accept an `onConvert` callback that hubs can override to show a "coming soon" toast instead of navigating.

## Label-to-control association (accessibility)

The two `<label>` elements ("From", "To") are visually positioned above their respective `<select>` elements but lack programmatic association. In React, this requires an `id` on the select and `htmlFor` on the label:

```tsx
<label htmlFor="format-from" className="...">From</label>
<select id="format-from" ...>
```

Without this, VoiceOver and NVDA announce the select as unlabelled. Since FormatPicker is rendered once per page currently but could be instantiated multiple times, the `id` values should use `useId()` from React 18 to avoid collisions.

## Header nav: Privacy link dropped

The old Header had five nav items: Image, PDF, Audio, Articles, Privacy (`/#privacy`). The new Header has five: Image, PDF, Audio, Text, Articles. The "Privacy" link pointed to the homepage privacy section — a trust anchor for a product whose brand is "your files are nobody's business." Dropping it removes the only top-nav path to that messaging.

Every hub page now includes privacy copy in the hero, which partially compensates, but users navigating from non-hub pages (articles, conversion pages) lose the one-click path to the privacy section. The footer links `/privacy` (the full policy page) but not `/#privacy` (the homepage trust section).

## ToolGrid semantic structure

ToolGrid renders tool names as `<p class="font-display text-lg">`. Since these appear inside a section with an `<h2>` above, the tool cards should use `<h3>` for their names. This gives screen reader users heading navigation into individual tools and improves the document outline for SEO (search engines treat heading nesting as a content-structure signal).
