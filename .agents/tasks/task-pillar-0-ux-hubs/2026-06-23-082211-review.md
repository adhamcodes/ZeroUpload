# Category UX hub pages with FormatPicker and ToolGrid (v2)

Four hub pages (/pdf, /image-converter, /audio-converter, /text) each surface a "From → To" format picker and a grid of category tools. The FormatPicker React island handles format selection and navigation; ToolGrid renders cards with coming-soon badges. Header and Footer are updated with all hub links; `/document-converter` redirects to `/pdf`. This pass verifies that the five issues raised in v1 were resolved.

All five v1 concerns are addressed: the /text hub's picker now shows a "coming soon" message instead of navigating to dead routes; select elements have proper `id`/`htmlFor` pairing via `useId()`; the Header restores the `/#privacy` link; same-format selection shows an inline hint; and ToolGrid cards use `<h3>` elements. Watch for: the `comingSoon` message text is hard-coded into FormatPicker rather than being customizable per-hub (possible, minor), and the Header now has six nav items which may crowd on smaller tablet viewports (possible, cosmetic).

**Verdict**: APPROVED

## High-level view

The FormatPicker gained a `comingSoon` boolean prop. When true, `handleConvert` short-circuits navigation and sets a React state message: "{FROM} to {TO} conversion is coming soon." The /text hub passes `comingSoon={true}`, so no 404 is possible. The other three hubs omit the prop and continue navigating normally.

Accessibility is resolved via `useId()` generating stable, collision-free IDs wired to `htmlFor`/`id` pairs. The same-format state now disables the button and shows "Choose different formats" in amber. The Header carries six nav items including the restored `/#privacy` link. ToolGrid uses `<h3>` for tool names in both card variants.

## ComingSoon prop design

The fix uses a boolean prop rather than a callback or custom message string. This means every hub with `comingSoon={true}` gets the same template:

```tsx
if (comingSoon) {
  setMessage(`${from.toUpperCase()} to ${to.toUpperCase()} conversion is coming soon.`);
  return;
}
```

The button stays enabled — the user gets click feedback rather than a mysteriously disabled control. If a second coming-soon hub needs different messaging in the future (say, a "premium only" gate), the prop would need to evolve into a string or callback. For now, with only /text using it, the boolean is sufficient.

## Nav density on smaller viewports

With six items at `gap-7`, the nav needs ~480px. The `sm:flex` breakpoint triggers at 640px, leaving only ~160px for the logo. On tablets in the 640–768px range, the links may visually crowd. Flexbox won't break (items won't wrap by default), but legibility suffers. A `md:flex` breakpoint or a tighter gap at `sm` (e.g., `sm:gap-4 md:gap-7`) would add breathing room.
