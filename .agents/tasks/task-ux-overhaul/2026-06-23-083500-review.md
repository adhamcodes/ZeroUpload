# UX overhaul: file-first converter flow, custom dropdowns, and design system depth

The change replaces native `<select>` elements with an accessible custom dropdown (`CustomSelect.tsx`), rewrites `Converter.tsx` into a file-first flow with per-file preview cards, batch cascade processing, and result "reward cards" showing before/after size comparisons. The design system gains depth/shadow elevation tokens, functional animations, texture utilities, and shifts dark mode to green-charcoal (oklch hue 165). Hub pages and the homepage receive surface wrappers and tightened spacing. The ambient `dropzone-glow` animation is fully removed.

**Watch for:** Progress bar positioning bug due to missing `relative` on `FileCard` root (confirmed). Files dropped during an active batch are silently discarded (confirmed). The `surface-elevated` utility sets its own `border-radius` which is immediately overridden by inline classes on every usage (likely).

## High-level view

The custom dropdown implements ARIA `combobox` + `listbox` semantics with full keyboard navigation, scroll-into-view, and mobile scroll-lock. JAWS has historically been inconsistent with `aria-activedescendant` on a combobox trigger when the listbox is a DOM sibling rather than a descendant — worth QA verification.

The Converter rewrite moves from a single `Status` enum to per-file state machine (`queued → converting → done | error`). A `convertingRef` guards against concurrent batches, but files dropped while a batch is running are appended to state and never processed — they stay "queued" permanently.

The `FileCard` progress bar uses `absolute` positioning but its root `<div>` lacks `position: relative`. The wrapper in the render loop provides containment, so the bar renders outside the card's rounded corners.

Dark mode tokens consistently use hue 165. The `--color-brass-soft` shifted from hue 70 to 165, reducing the warm-gold accent in dark mode — intentional or not, it changes the feel. The elevation utility `.surface-elevated` sets `border-radius: var(--radius-md)` in CSS, but every Astro usage overrides it with `rounded-[var(--radius-xl)]`.

No engine files (`src/lib/`) were modified. Ambient glow is fully excised.

1. **Progress bar escapes card bounds** — `FileCard` root div is not `position: relative`, so the `absolute`-positioned progress bar clips to the outer wrapper rather than the card itself. Add `relative` to the FileCard root's class list.

## Batch processing race when files are added mid-conversion

The `handleFiles` function guards batch start with `if (!convertingRef.current)`. When a batch is already running and the user drops additional files, the code creates file entries (cards appear as "queued") but skips `processBatch`. Those entries stay in "queued" state permanently until the user resets.

```tsx
// Current (broken):
if (!convertingRef.current) {
  convertingRef.current = true;
  setIsConverting(true);
  await processBatch(newEntries, d);
}
// newEntries silently orphaned if convertingRef.current === true
```

The drop zone's `onClick` handler no longer gates on `isConverting` (previously it was `!working && inputRef.current?.click()`), making it easy for users to trigger this path. Fix: maintain a ref holding pending entries, and at the end of `processBatch`, check if more arrived and drain them.

## FileCard progress bar positioning

```tsx
<div className={["animate-slide-up flex items-center gap-4 rounded-[...] ...", ...].join(" ")}>
  ...
  {entry.status === "converting" && (
    <div className="absolute bottom-0 left-0 right-0 h-0.5 overflow-hidden rounded-b-[var(--radius-xl)]">
      <div className="animate-progress-fill h-full bg-accent" />
    </div>
  )}
</div>
```

The containing block for `absolute` is the nearest positioned ancestor — the `<div className="relative">` wrapper in the parent loop, not the card itself. The progress bar won't clip to the card's `rounded-b-[...]`. Adding `relative overflow-hidden` to the FileCard root fixes both positioning and corner clipping.

## ARIA on CustomSelect

The implementation follows WAI-ARIA's select-only combobox pattern. One subtlety: `aria-activedescendant` is set on the button (role `combobox`) while the listbox is a DOM sibling. NVDA and VoiceOver handle this correctly, but JAWS has historically been inconsistent with this arrangement. Not blocking, but worth manual screen-reader QA.

## Dark mode `--color-brass-soft` hue shift

`--color-brass-soft` moved from hue 70 (warm gold family) to hue 165 (green-charcoal). This makes brass-soft indistinguishable from accent-soft in dark mode. If the intent is to preserve the warm/cool contrast between brass and accent tokens, this should stay at hue 70 with a lower lightness.

## Redundant border-radius on elevation utilities

`.surface-elevated` declares `border-radius: var(--radius-md)` and `.surface-raised` declares `border-radius: var(--radius-xl)`. Every Astro usage of `surface-elevated` adds `rounded-[var(--radius-xl)]`, overriding the CSS declaration. Either align the utility to `--radius-xl` or remove the inline overrides.
